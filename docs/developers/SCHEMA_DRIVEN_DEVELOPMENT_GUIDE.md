# PixEagle Schema And Extension Guide

PixEagle uses schemas for public tracker data, follower command fields, runtime
configuration, and dashboard metadata. Schemas do not replace implementation
registration: a new tracker or follower needs both a typed contract and a
registered implementation.

## Contract Authorities

| Concern | Authority | Purpose |
|---|---|---|
| Follower command fields, units, profiles, and control types | `configs/follower_commands.yaml` | Defines the complete command snapshot accepted from each follower |
| Tracker output shapes and tracker/follower compatibility | `configs/tracker_schemas.yaml` | Defines tracker sample validation and the current compatibility matrix |
| Runtime configuration defaults | `configs/config_default.yaml` | Defines checked-in operator defaults |
| Runtime configuration API/schema | `configs/config_schema.yaml` | Generated by `scripts/generate_schema.py`; do not hand-edit |
| Follower implementations | `FollowerFactory` in `src/classes/follower.py` | Maps profile names to concrete classes |
| Tracker implementations | `TRACKER_REGISTRY` in `src/classes/trackers/tracker_factory.py` | Maps algorithm names to concrete classes |
| Tracker output enum | `TrackerDataType` in `src/classes/tracker_output.py` | Defines data-type values used by Python callers |

The factory registry and enum remain explicit Python contracts. Adding YAML
alone does not make an implementation available.

## Follower Command Contract

The maintained command fields are:

| Field | Unit | Direction or range |
|---|---|---|
| `vel_body_fwd` | m/s | Positive forward |
| `vel_body_right` | m/s | Positive right |
| `vel_body_down` | m/s | Positive down |
| `yawspeed_deg_s` | deg/s | Positive clockwise |
| `rollspeed_deg_s` | deg/s | MAVSDK attitude-rate convention |
| `pitchspeed_deg_s` | deg/s | MAVSDK attitude-rate convention |
| `thrust` | normalized | 0.0 through 1.0 |

The maintained control types are `velocity_body_offboard` and
`attitude_rate`. Removed field names and control types are rejected; they are
not runtime aliases.

Follower code publishes one atomic snapshot:

```python
if not self.set_command_fields(
    {
        "vel_body_fwd": forward_m_s,
        "vel_body_right": right_m_s,
        "vel_body_down": down_m_s,
        "yawspeed_deg_s": yaw_speed_deg_s,
    },
    reason="my_follower_normal_tracking",
):
    raise RuntimeError("command intent was rejected")
```

`set_command_fields()` validates the entire declared profile before changing
live state. Do not issue MAVSDK calls from a follower. `OffboardCommander` owns
application-level command publication on the flight event loop.

Velocity and angular-rate envelopes come from `Safety.GlobalLimits` and valid
tightening `Safety.FollowerOverrides`. Do not duplicate operational safety
limits in `follower_commands.yaml`.

## Tracker Output Contract

Trackers return `TrackerOutput` instances. A position tracker normally emits a
sample like this:

```python
return TrackerOutput(
    data_type=TrackerDataType.POSITION_2D,
    timestamp=time.time(),
    tracking_active=True,
    tracker_id="my_tracker",
    position_2d=(x_normalized, y_normalized),
    confidence=confidence,
    raw_data={"usable_for_following": True},
)
```

The output must satisfy `configs/tracker_schemas.yaml`. Tracking state alone is
not enough for flight control: freshness and `usable_for_following` are checked
at the AppController/follower boundary. A tracker must publish explicit stale or
inactive truth when measurements are no longer usable.

## Add A Follower

1. Choose a unique lower-snake-case profile name.
2. Add its complete profile to `configs/follower_commands.yaml` using only
   maintained fields and control types.
3. Implement a `BaseFollower` subclass under `src/classes/followers/`.
4. Register it in `FollowerFactory._follower_registry`.
5. Add the profile to `FollowerType` in `src/classes/follower_types.py`.
6. Add tracker compatibility to `configs/tracker_schemas.yaml` and declare the
   profile's required/optional tracker data in `follower_commands.yaml`.
7. Add user-tunable parameters to `configs/config_default.yaml`, regenerate the
   schema, and add validation/retirement entries when replacing an old key.
8. Add unit tests for normal commands, target loss, stale input, non-finite
   values, safety limits, profile/registry consistency, and command-frame signs.
9. Add a reference page under `docs/followers/02-reference/`.

Minimal implementation shape:

```python
class MyFollower(BaseFollower):
    def __init__(self, px4_controller, initial_target_coords):
        super().__init__(px4_controller, "my_follower")

    def calculate_control_commands(self, tracker_data: TrackerOutput) -> None:
        target = self.extract_target_coordinates(tracker_data)
        if target is None:
            raise ValueError("position data is required")

        self.set_command_fields(
            {
                "vel_body_fwd": 0.0,
                "vel_body_right": lateral_controller(target[0]),
                "vel_body_down": vertical_controller(target[1]),
                "yawspeed_deg_s": 0.0,
            },
            reason="my_follower_normal_tracking",
        )

    def follow_target(self, tracker_data: TrackerOutput) -> bool:
        if not self.validate_tracker_compatibility(tracker_data):
            return False
        self.calculate_control_commands(tracker_data)
        return True
```

Each follower must define an explicit inactive/target-loss policy. The default
base behavior rejects inactive samples. A follower may opt in only when it
publishes a bounded hold, stop, coast, or otherwise reviewed target-loss intent.

## Add A Tracker

1. Choose a unique algorithm key and implement the `BaseTracker` contract under
   `src/classes/trackers/`.
2. Register the class in `TRACKER_REGISTRY`.
3. Reuse an existing `TrackerDataType` when its data contract matches. For a new
   data type, update the Python enum and `tracker_data_types` schema together.
4. Add tracker metadata under `tracker_types` in
   `configs/tracker_schemas.yaml`.
5. Update follower compatibility only where the implementation and current
   sample fields support it.
6. Add tests for initialization, selection geometry, confidence/freshness,
   target loss, restart behavior, and catalog/registry agreement.
7. Add setup guidance for optional native dependencies. Missing optional
   packages must produce an actionable readiness error, not a late import crash.

## Configuration Changes

Do not edit `configs/config_schema.yaml` manually. After changing defaults or
config metadata, run:

```bash
.venv/bin/python scripts/generate_schema.py
bash scripts/check_schema.sh
```

When removing or renaming a key, add an exact retirement rule to
`configs/config_retirements.yaml`. A retired key must not remain as an active
runtime alias.

## API Surfaces

New public APIs use typed `/api/v1/...` routes. Relevant maintained routes
include:

- `GET /api/v1/tracking/catalog`
- `GET /api/v1/tracking/runtime-status`
- `GET /api/v1/tracking/telemetry`
- `POST /api/v1/actions/tracker-switch`
- `POST /api/v1/actions/tracker-restart`
- `GET /api/v1/following/status`
- `GET /api/v1/following/telemetry`

The current follower profile/schema catalog remains exposed through tracked
compatibility routes under `/api/follower/...` while its typed v1 replacement is
completed. Do not add new behavior to those compatibility routes.

MCP and AI-agent integrations must consume the same typed API/state contracts;
do not create a second automation-only command path.

## Required Validation

At minimum, run:

```bash
PYTHONPATH=src .venv/bin/pytest -q \
  tests/unit/followers/test_config_consistency.py \
  tests/unit/drone_interface/test_setpoint_handler.py \
  tests/unit/test_base_follower.py
bash scripts/check_schema.sh
```

Also run the concrete follower/tracker tests and the AppController Offboard
safety suite for behavior changes. Unit and mock tests do not prove SITL, HIL,
or real-aircraft behavior; record those separately with exact versions,
configuration, logs, and evidence artifacts.
